# 复杂项目与迭代式开发的管理方法

本文档面向「从简单项目过渡到多模块、可迭代产品」的开发者，介绍如何用 Git 和简单流程把需求、反馈和代码变更管起来，避免「东一下西一下」。

---

## 一、核心原则（先建立心智）

1. **小步提交、可回滚**  
   每完成一个小功能或修掉一类问题就 commit，方便随时回退或对比。

2. **需求与实现分离**  
   先想清楚「要做什么、做到什么程度」，再动手写；避免边写边改范围。

3. **反馈要可执行**  
   把「感觉不对」变成「具体哪里、期望怎样、优先级如何」，便于排期和验收。

4. **单点维护配置与约定**  
   像大类/小类这种会扩展的，放在一个配置文件里（如 `src/lib/categories.ts`），避免到处硬编码。

---

## 二、用 Git 做「可控迭代」

### 2.1 分支习惯（按你团队规模选一种即可）

- **单人 / 小项目**  
  - `main` 始终是可运行版本。  
  - 新功能或大改：从 `main` 拉 `feature/xxx`，做完再合并回 `main`。  
  - 修 bug：可直接在 `main` 上小步提交，或拉 `fix/xxx` 再合并。

- **多人或希望更稳**  
  - `main` = 稳定、可发布。  
  - 开发在 `develop` 或各自 `feature/xxx`，测试通过后再合并到 `main`。  
  - 修线上问题可拉 `hotfix/xxx`，修完合并回 `main`（和 `develop`）。

### 2.2 提交信息（commit message）

建议格式：**类型 + 简短说明**，必要时加一句「为什么」。

- `feat: 新增按大类/小类配置，表单联动`  
- `fix: 份额修改保存后表格不刷新`  
- `refactor: 大类顺序与颜色抽到 categories.ts`  
- `docs: 补充盈亏列数据来源说明`

这样以后用 `git log`、`git blame` 都能快速看懂每次改动的目的。

### 2.3 何时打 tag

- 每做完一个你心目中的「小版本」（例如：分类改版、盈亏列稳定、首次给他人用），在 `main` 上打 tag，如 `v0.2.0`。  
- 方便以后对比「v0.1 和 v0.2 之间改了什么」，也方便回滚到某个版本。

## 三、如何「有组织地」提反馈与需求

你提到：看到 prototype 后有很多问题，但反馈有点散。可以按下面方式收拢。

### 3.1 按「类型」分（方便排优先级）

| 类型 | 含义 | 举例 |
|------|------|------|
| **Bug** | 与预期不符、报错、数据错 | 修改份额后界面无变化、本月盈亏占两行 |
| **体验/UX** | 能用但难用、不直观 | 查代码要手动点、不知道三月盈亏从哪来 |
| **功能增强** | 希望多做什么 | 根据名称自动判断类型与风险、自动查代码 |
| **配置/约定** | 规则要统一、可扩展 | 大类小类按「现金/债券/商品/权益」且可扩展 |

同一轮反馈里，可以简单标一下：这是 bug / 体验 / 功能 / 配置。

### 3.2 按「模块」分（方便改代码时聚焦）

- 表格与总览  
- 产品/流水/净值/快照  
- 分类与目标占比  
- 自动化（查代码、刷新净值、三月/六月盈亏）  
- 文档与说明  

提反馈时带一句「属于哪一块」，后续改代码和做 commit 都会更集中。

### 3.3 一条反馈尽量包含三件事

1. **现象**：现在是什么样（或哪一步会出问题）。  
2. **期望**：你希望变成什么样（或希望多出什么信息）。  
3. **优先级**（可选）：必须马上修 / 本周 / 有空再做。

示例：  
「表格里本月盈亏这一列会换行（现象），希望和另外几列一样单行显示（期望），属于表格展示，优先级高（可选）。」

这样 AI 或同事能直接给出针对性的改法，而不是猜。

## 四、建议的「轻量流程」（不依赖复杂工具）

### 4.1 需求/待办从哪记

- **方式 A**：在项目里建一个 `docs/BACKLOG.md`（或 `TODO.md`），按「Bug / 体验 / 功能 / 配置」或按模块列清单，每条一行，前面加 `[ ]`，做完改成 `[x]`。  
- **方式 B**：用 GitHub/GitLab 的 Issue，一条反馈一个 issue，用 label 区分类型和模块。  
- **方式 C**：继续在对话里提，但每次集中 3～5 条，并按上面「类型 + 模块 + 现象/期望」写清楚。

选一种你愿意坚持的即可，关键是**有地方可回溯**，而不是散在聊天里。

### 4.2 每轮迭代可以怎么做

1. **收一轮反馈**：把「东一下西一下」的零散点，按上面方式整理成几条「可执行项」。  
2. **排个顺序**：先 bug，再体验，再功能；或先做你最在意的 1～2 个模块。  
3. **小步做**：每完成 1～2 条就 commit，必要时拉 `feature/xxx`。  
4. **做完简单自测**：例如「改份额 → 保存 → 刷新是否正常」「本月盈亏是否还占两行」。  
5. **更新文档**：在 `PROJECT-PROGRESS.md` 或 `CHANGELOG.md` 里记一笔「本轮做了啥」，方便下次接着迭代。

### 4.3 什么时候值得「停一下」做整理

- 同一块代码改了很多次、已经很难读；  
- 分类、配置、常量散落在多处；  
- 你发现「要加一个小类/新模块」却不知道从哪改起。

这时可以专门做一轮「小 refactor」：抽配置（如 categories）、统一命名、拆大文件，再 commit 为 `refactor: xxx`，然后再继续加功能。

## 五、和 AI 协作时怎么更高效

- **一次给足上下文**：例如「按我现在的分类方式：现金(现金/日元/美元)、债券(债券/债+股)…」比零散说「改一下分类」更省来回。  
- **指明「单点维护」**：例如「以后新增大类小类只改一个地方，别到处写死」，AI 就会倾向于抽成 `categories.ts` 这种配置。  
- **验收标准简单写清**：例如「新增产品时选债券，细分里只能选债券或债+股」，改完你可以按这个 checklist 点一遍。

## 六、本项目的相关文件（可当作模板）

| 文件 | 用途 |
|------|------|
| `docs/PROJECT-PROGRESS.md` | 当前进度、已完成、后续计划（给 AI 或自己续写用） |
| `docs/BACKLOG.md` 或 Issue | 待办与反馈清单（你可自建） |
| `src/lib/categories.ts` | 大类/小类单点配置，新类别在此扩展 |
| `docs/pnl-columns-logic.md` | 某块逻辑的说明（盈亏列数据来源） |
| `docs/project-management-guide.md` | 本文档，项目管理与反馈方法 |

---

总结：用 **Git 小步提交 + 明确的需求/反馈写法 + 一个固定的待办清单（或 Issue）**，就能在「不断迭代、不断加功能」的过程中保持可控；你补充的「按我现在的分类方式来分大类小类、新类别以此类推」已经用 `categories.ts` 单点维护落地，后续新类别只要改那一处即可。
